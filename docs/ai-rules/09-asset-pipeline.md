# 09 — Asset pipeline (sprites e mapas)

> Como arte (PNG do Aseprite) e mapas (JSON do Tiled) viram bytes
> no firmware. Decisão fechada: **pré-compilar tudo**, zero parsing
> em runtime.

## Princípios

1. **PNG/JSON são formatos de edição**, ficam no `assets/` do repo,
   nunca no firmware.
2. **Pipeline de build** (script Python + ferramenta LVGL) converte
   para formatos consumíveis: bin RGB565 (sprites) e `const` arrays C
   (mapas).
3. **Firmware lê bytes prontos**, sem parser, sem heap dinâmico de
   carregamento.

Resultado: zero fragmentação de heap, latência de carga = velocidade
da NAND, mapas validáveis em compile-time.

## Sprites

### Edição: Aseprite → PNG

```
assets/
└── entities/
    ├── player/
    │   ├── player_idle_down.aseprite
    │   ├── player_idle_down.png    (export do .aseprite)
    │   ├── player_walk_down_01.png
    │   └── ...
    ├── npc/
    ├── furniture/
    │   ├── chair_01.png
    │   ├── table_01.png
    │   └── ...
    └── props/
```

PNG **só vive aqui no repo**, nunca no firmware. Versionar no git
quando estável.

### Conversão: PNG → bin RGB565

Ferramenta: **`lvgl-image-converter`** (consistente com pipeline já
estabelecida no projeto). Roda como Python script no host.

Comando (placeholder — script wrapper em `tools/convert_sprites.py`):

```bash
python tools/convert_sprites.py \
    --input  assets/entities/ \
    --output build/sprites/ \
    --format rgb565_be   # big-endian para match com display_hal
```

Output: `build/sprites/player_idle_down.bin` etc. Cada arquivo é
RGB565 puro (sem header — tamanho conhecido pelo nome da entity).

### Upload: bin → NAND

Comando manual via `recovery` USB CDC:

```bash
python tools/upload_assets.py --port COM17 --dir build/sprites/
```

Script ainda **não existe** — bloqueador B2 em
[[../../CyberGameCore/_AGENT/OPEN_QUESTIONS|OPEN_QUESTIONS]]. Hoje
fluxo é manual via PING/PONG (que só testa conectividade).

Quando B2 estiver feito: `recovery` aceita comandos `PUT category id
bytes`, `GET category id`, `LIST`.

### Consumo: NAND → LVGL

Em runtime, `asset_loader` (já existe em `components/assets/`) abstrai:

```c
const lv_image_dsc_t *asset_loader_get(const char *name);
```

Internamente: lê da NAND via `asset_store`, monta `lv_image_dsc_t` com
ponteiro para buffer em SRAM (sprites pequenos — 32×48 RGB565 = 3 KB)
ou em DMA-capable region.

Entity recebe `lv_image_dsc_t` e seta com `lv_image_set_src`.

## Mapas

### Edição: Tiled → JSON

Editor: **Tiled** (https://www.mapeditor.org/) — formato `.tmj` (JSON
flavor de Tiled). Exporta como JSON puro.

```
assets/
└── maps/
    ├── room_recepcao.tmj          (editado no Tiled)
    ├── room_recepcao.json         (export JSON do Tiled)
    ├── room_empresa.json
    └── ...
```

Formato Tiled inclui layers, tile IDs, objects (entities), properties.
Não precisamos de toda a richness — script extrai só o que importa.

### Conversão: JSON → arrays C

Script: `tools/maps_to_c.py` (a criar).

Input: `assets/maps/room_recepcao.json`
Output: `components/gamestate/maps/map_recepcao.c` + `.h`

Exemplo do `.c` gerado:

```c
// AUTO-GENERATED by tools/maps_to_c.py — DO NOT EDIT
// Source: assets/maps/room_recepcao.json
// Generated: 2026-05-27T11:30 by user@host

#include "map_data.h"
#include "entity.h"

static const uint16_t s_ground_recepcao[MAP_TILES_W * MAP_TILES_H] = {
    1, 1, 1, 1, 1, /* ... 30 * 20 = 600 valores */
};

static const uint16_t s_walls_recepcao[MAP_TILES_W * MAP_TILES_H] = {
    0, 0, 12, 12, 12, /* ... */
};

static const uint16_t s_overlay_recepcao[MAP_TILES_W * MAP_TILES_H] = {
    0, 0, 0, 0, 0, /* ... */
};

static const uint8_t s_collision_recepcao[MAP_TILES_W * MAP_TILES_H] = {
    0, 0, 1, 1, 1, /* 1 = TILE_SOLID */
};

/* Entities iniciais da sala. Carregadas em entity_pool. */
static const map_entity_init_t s_entities_recepcao[] = {
    { .type = ENTITY_TYPE_NPC,       .x = 120, .y = 160, .sprite = "npc_recepcionista",
      .flags = ENTITY_FLAG_SOLID | ENTITY_FLAG_INTERACTABLE | ENTITY_FLAG_YSORTED,
      .collision_offset_x = -7, .collision_offset_y = -8,
      .collision_w = 14, .collision_h = 8, .sort_offset_y = 0 },
    { .type = ENTITY_TYPE_FURNITURE, .x = 80,  .y = 96,  .sprite = "balcao_01",
      .flags = ENTITY_FLAG_SOLID | ENTITY_FLAG_YSORTED,
      .collision_offset_x = -16, .collision_offset_y = -8,
      .collision_w = 32, .collision_h = 8 },
    /* ... */
};

/* Triggers (zonas invisiveis). */
static const map_trigger_init_t s_triggers_recepcao[] = {
    { .id = TRIGGER_ID_PORTA_EMPRESA, .x = 400, .y = 280, .w = 32, .h = 16,
      .event = FSM_EVT_TROCAR_SALA_EMPRESA },
};

const map_data_t MAP_RECEPCAO = {
    .name        = "recepcao",
    .ground      = s_ground_recepcao,
    .walls       = s_walls_recepcao,
    .overlay     = s_overlay_recepcao,
    .collision   = s_collision_recepcao,
    .entities    = s_entities_recepcao,
    .entity_count = sizeof(s_entities_recepcao) / sizeof(s_entities_recepcao[0]),
    .triggers    = s_triggers_recepcao,
    .trigger_count = sizeof(s_triggers_recepcao) / sizeof(s_triggers_recepcao[0]),
};
```

`.h` declara `extern const map_data_t MAP_RECEPCAO;`.

### Consumo: array C → entity_pool

```c
// components/gamestate/map_loader.c
void map_load(const map_data_t *map)
{
    tilemap_load(map->ground, map->walls, map->overlay, map->collision);
    entity_pool_clear();
    for (size_t i = 0; i < map->entity_count; ++i) {
        const map_entity_init_t *init = &map->entities[i];
        entity_create_from_init(init);   /* aloca pool, sprite_get, cria lv_obj */
    }
    trigger_load(map->triggers, map->trigger_count);
}
```

Zero parsing. Tudo direto da flash (constante linkada na imagem).

## Tamanho na imagem

Por sala:
- 3 layers × 30 × 20 × 2 bytes = **3.6 KB** (ground/walls/overlay)
- 1 layer × 30 × 20 × 1 byte = **600 B** (collision)
- 30 entities × ~40 bytes = **1.2 KB**
- 10 triggers × ~16 bytes = **160 B**
- **Total: ~5.5 KB por sala**

10 salas no MVP = **55 KB** no .bin. Aceitável (binário hoje ~570 KB,
86% partição livre).

Sprites ficam na NAND, não no .bin do firmware.

## Quando o gerador roda

### Manual (preferido por hora):

```bash
python tools/maps_to_c.py assets/maps/*.json
```

Antes de cada commit que mexe em mapa. Editar Tiled, salvar JSON,
rodar script, build.

### Automático (futuro):

Hook do CMake: regenera `.c` se `.json` for mais novo. Custo: ~1s por
mapa pequeno. Provavelmente atrasa o build inicial sem ganho real.
Considerar quando virar dor.

## Regras

### R9.1 — Editar `.c` gerado é proibido

`map_*.c` em `components/gamestate/maps/` tem header
`// AUTO-GENERATED — DO NOT EDIT`. Edit é descartado no próximo run do
script. Para mudar mapa: editar `.json`, rodar script.

### R9.2 — `.json` do mapa é versionado

Commit do `.json` E do `.c` resultante juntos. Permite revisão de PR
sem rodar o script.

### R9.3 — Sprite path como string em entity_init

Em `map_entity_init_t.sprite` é o **nome lógico** do sprite (`"chair_01"`).
`asset_loader_get` mapeia isso para o arquivo bin na NAND. Sem extensão.

### R9.4 — Tile IDs >= 1 (zero é vazio)

ID 0 reservado para "sem tile". IDs reais começam em 1. Atlas de tiles
mantido em `assets/tiles/` e identificado por ID consistente entre
mapas.

### R9.5 — Scripts vivem em `tools/`

`tools/convert_sprites.py`, `tools/maps_to_c.py`, `tools/upload_assets.py`.
Python 3. Sem dependências exóticas (Pillow para sprites, json builtin
para mapas).

### R9.6 — Output do script tem timestamp + path-source

Comentário no topo do `.c` gerado inclui timestamp e arquivo original.
Facilita rastreio em git blame quando algo quebra.

### R9.7 — Mapas carregados via ponteiro `const`

`map_load(&MAP_RECEPCAO)`, não `map_load("recepcao")`. Sem
indireção por string em runtime.

## Anti-padrões

- ❌ `cJSON_Parse` em runtime no caminho de troca de sala.
- ❌ Mapa hardcoded inline em `screen_*.c`.
- ❌ Editar `.c` gerado e commitar sem editar `.json`.
- ❌ Sprite PNG carregado em runtime via decoder.
- ❌ Cada sala alocando heap dinâmico para o tilemap.
