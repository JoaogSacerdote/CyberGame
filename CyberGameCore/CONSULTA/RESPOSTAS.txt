Para garantir que a sobreposição de imagens no mapa 2D top-down funcione corretamente e sem estourar a memória RAM do ESP32-S3, a construção visual no LVGL obedecerá rigorosamente à técnica de Z-Ordering (Ordem de Profundidade), estruturada em 5 camadas lógicas hierárquicas:

Camada 0 (Background / Chão): Elementos 100% estáticos e sem colisão física (ex: texturas de piso, carpetes). Renderizados no fundo. O personagem sempre sobrepõe esta camada.

Camada 1 (Midground / Obstáculos): Elementos estáticos com colisão física ativa (ex: mesas de escritório, paredes, servidores). O personagem colide e não os atravessa.

Camada 2 (Entidades Dinâmicas): Elementos com movimento e atualização de tela constante (ex: Sprite do Analista, alertas visuais de ataque). Possuem colisão dinâmica entre si e contra a Camada 1.

Camada 3 (Foreground / Teto): Elementos estáticos de sobreposição visual para criar sensação de profundidade (ex: molduras superiores de portas, infraestrutura de teto). Não possuem colisão; o personagem caminha por baixo deles.

Camada 4 (UI / HUD / Terminal): Camada de interface absoluta. Ignora a câmera do cenário. Contém o Relógio, Vidas e a Tela do Terminal. É ancorada no topo da hierarquia visual, sobrepondo todas as camadas anteriores.

Nota de Engenharia (LVGL): O sistema não aloca buffers de vídeo individuais para cada camada (o que inviabilizaria o uso do microcontrolador). A profundidade é tratada puramente pela ordem de criação dos objetos na árvore (Parent/Child). A Camada 4 (HUD) será sempre desenhada por último ou forçada para a frente utilizando a função nativa lv_obj_move_foreground().

🧠 BLOCO 6: Arquitetura de Software e Tasks (FreeRTOS)
18. Divisão de Tarefas (Task Management)
O código não rodará num único loop (como num Arduino básico). Ele será dividido em três rotinas independentes para garantir a ortogonalidade exigida no projeto:

1. Task UI / LVGL (Prioridade Alta): * Dedicada exclusivamente a redesenhar as áreas sujas do ecrã e gerir as animações usando os buffers e o DMA. Não faz cálculos de jogo.

2. Task GameEngine (Prioridade Média): * É o cérebro da operação. Executa a Máquina de Estados (FSM) que aprovámos no Bloco 4, os timers do relógio do expediente e aplica as penalizações ou bónus do Bloco 5.

3. Task Hardware (Prioridade Alta / Orientada a Interrupção): * Fica adormecida na maior parte do tempo. Acorda apenas para ler os inputs físicos do joystick/botões e o módulo NFC.

19. Comunicação Segura (IPC - Inter-Process Communication)
Para que uma rotina não atropele a outra, a comunicação entre elas será blindada:

Filas (Queues): O joystick e os botões não mudam o personagem diretamente. A Task Hardware empacota a direção (ex: CIMA) num pacote de dados e atira para a "Fila de Comandos". A Task GameEngine lê essa fila e decide se o personagem pode andar ou se está em estado de Action Lock.

Mutex (Cadeado do Ecrã): A biblioteca LVGL tem de ser protegida. Quando a Task GameEngine precisar de mudar a cor de um servidor para vermelho, ela adquire um Mutex, faz a alteração em segurança, e liberta o Mutex para que a Task UI possa desenhar.

20. O Gatilho do NFC (A Tua Correção)
Refletindo exatamente o que definiste no Bloco 3 e 4:

A Task Hardware mantém a rotina I2C/SPI do chip PN532 completamente suspensa.

Quando a Task GameEngine muda a FSM para o estado de Terminal de Ataque, ela levanta uma Event Flag (Sinalizador).

A Task Hardware vê este sinal, energiza o PN532 e fica à escuta da carta. Mal a leitura é validada ou o jogador sai do Terminal, o sinal cai e o leitor é desligado de imediato.

5. Arquitetura FreeRTOS e Afinidade de Núcleos (ESP32-S3)
Para respeitar o Princípio da Ortogonalidade e evitar engasgos gráficos, as tarefas devem ser fixadas nos núcleos da seguinte forma:

Core 1 (Application Core): Exclusivo para a Task UI / LVGL. Responsável por todo o processamento gráfico, animações e envio de dados via DMA para o display.

Core 0 (Protocol Core): Responsável pela Task GameEngine (lógica da FSM, timers, regras de pontuação) e pela Task Hardware (leitura de joystick, botões e acionamento do NFC).

6. Renderização e Camadas (Z-Ordering)
O motor gráfico deve obrigatoriamente seguir a estrutura de 5 camadas lógicas (0: Chão, 1: Obstáculos/Colisão, 2: Personagem/Entidades, 3: Teto/Sobreposição, 4: HUD/Terminal) para garantir profundidade visual e impedir que elementos do cenário cubram as informações vitais da interface (HUD).
